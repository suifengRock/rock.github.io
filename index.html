bytes.Buffer panic out of memory

发表于 2017-12-09 | 分类于 Golang

在使用bytes.Buffer作读写缓冲数据时，需要注意几个问题，之前踩了大坑，被坑的不要不要的。
数据缓冲的原理就是，把数据都缓存到内存里面，后续的读写都从缓存里面直接操作。这里面要注意
几个问题：自己这个buffer最多会占用多少内存？什么时候释放？会不会影响其他程序？
宿主机资源够不够？等等，这些做的时候要考虑清楚，不然上线后一不小心就会panic.
下面说一下主要会引起的panic和解决办法。

bytes.Buffer: too large

在使用bytes.Buffer读写时，函数说明里面都有一段if becomes too large, will panic ErrTooLarge.意思就是如果缓冲了太多数据，超过了限制的话，就非常暴力的直接panic。一开始不理解为什么要那么暴力，直到看到源码

buffer缓冲的最大限制就是maxInt这个数，有9223372036854775807 byte这么大,换算到后面要用PB作单位。。。这么一看，可以理解为什么那么暴力了.缓冲那么超级大量数据，要么就是失误，要么就是整体方案不对，所以选择直接panic暴露出来。这个问题我们不用太在意，现实上根本(极极极极小概率)不会遇到，毕竟现实中内存有超过maxInt的使用场景基本没有。

out of memory

相比上面的panic,我们更应注意这个panic。顾名思义引起这个panic有几个错误，内存溢出或者宿主内存资源不足，资源耗尽。在使用buffer的时候就要注意，缓冲数据会不会过大？要提前做好限制。因为一般测试都是比较数据比较小，这个问题不会暴露，只有等到了线上时，资源被耗尽情况下，才会panic.又因为资源耗尽，没有做好隔离的话，会引起整个业务的崩溃。
解决的办法只有一个，也是必须要做的，做好缓冲数据的容量控制。

最后提一下:
上面只是提到buffer会panic这个问题，翻下源码，你会发现buffer底层使用的是[]byte这个slice，理论上来说任何一个slice不做控制的话，都会报这个panic.原因就是slice扩容的时候，是扩容2倍当前容量，如果不加控制，资源肯定会无休止的耗尽，最后panic。
要清楚的知道自己的边界，了解自我，做人如此，程序也如此。

Stay Real !!!~

golang 工程目录和gopath规范

发表于 2017-07-29 | 分类于 Golang

golang是一门工程性很强的语言。从一开始就约定了很多规范：语法，代码格式，gobin,gopath，目录结构等
没有正确理解和正确使用的话，就会觉得golang很繁琐，而且会有很多奇奇怪怪的坑。

golang目录结构

golang工程项目必须有三个目录 bin, pkg,src

gopath规范

根据官方的最佳推荐，gopath其实只要全局设定一个就可以，可以把它理解为工作区间。
go项目都放在，统一的工作区间的src目录里面。不需要每一个项目都去设置它的gopath.
每个项目独立的第三方依赖可以放在各自项目目录下的vendor里面。这样每个项目都不会相互影响。
提交代码只要提交src下的项目代码就可以，不需要把bin,pkg等其他提交上去。目前开源项目都遵循这种规范。

这样有个问题：编辑器里面的目录栏会有多个项目一起展现，看起来不干净，找代码比较麻烦；而且go的命令都是在gopath下执行，但git一些命令又在代码目录下执行，要来回切换目录。
我的解决方法是： 引入make工具来管理简化各种命令

这样就可以按自己需要制定和简化命令操作，大大方便搬砖。

git - 团队多人协作最佳实践

发表于 2017-07-16 | 分类于 Git

git是一个伟大的发明。是一个很好的分布式版本控制工具。
但在团队没有一个很好地规范的情况，会产生很多不必要的冲突和麻烦。
尤其是git分支管理维护，没有一个很好的，团队一致的流程，那将是一个灾难。

git分支管理

分支(branch)在git中是一个非常重要的概念。
对于分支管理策略，大家普遍遵从git-flow流程。

结合图片，简单介绍一下不同分支的职责。

1.Production分支(Master)

这个分支是发布到生产环境的代码，这个分支只能从其他分支合并，不能在这个分支直接修改。

2.Develop分支

这个分支是主开发分支，包含所有要发布到下一个Release的代码，这个主要合并自其他分支，比如Feature分支。

3.Feature分支

Feature 分支主要用来开发一个新的功能，一旦开发完成，合并回Develop分支，并且进入下一个Release，Feature分支可以选择删除或者保留。

4.Release分支

当需要发布一个新Release的时候，基于Develop分支创建一个Release分支，Release分支在测试过程中可能会修改，完成Release后，合并到Master和Develop分支。

5.Hotfix分支

当在Production发现新的Bug时候，需要创建一个Hotfix分支, 完成Hotfix后，合并回Master和Develop分支，所以Hotfix的改动会进入下一个Release。

Git Flow使用原则

  • Master分支是线上稳定分支，Release通常用作测试分支，Develop分支是开发应用的主分支
  • 所有的功能开发都在Feature分支进行，然后合并到Develop分支
  • Release分支发布后出现问题，直接在Release分支修改，避免Develop分支代码污染

git团队提交流程

团队中保持统一的提交流程，有利于减少不必要的冲突和异常问题，减少误操作，便于新成员融入。
在团队协作中，通常用一个git仓库中心，便于统一管理和存储，一般团内内部自建gitlib，或者选用公共平台私有团队仓库(github,bicketbucketd等).

下面分享一个，个人经过这几年来总结而来的提交流程：

当已经把更改的文件add，并且commit本地，想合并提交到团队共同开发分支时,假设目前自己本地的开发分支feature/new ,共同开发分支为develop。

  • git check develop
  • git pull origin develop
  • git checkout feature/new
  • git rebase develop
  • git push origin feature/new
  • 在线上仓库中心开启一个MR(merge request)或者PR(pull request), 把新提交feature/new合并到develop

此流程可以解决git团队协作中几大问题：
1：提交导致分支网络一片混乱
2：多人同时提交，代码被覆盖问题
3：code review 和合并权限管理问题

《使用 git rebase 避免無謂的 merge》

最后

强烈建议使用git-flow这个插件,省事省力省心！！！

工具终究是个死物，效果主要取决用的主人。清楚自己每一步都在干什么，这个才是最关键的。

go vendor机制与注意点

发表于 2017-04-15 | 分类于 Golang

go自1.5后正式推出vendor用于管理依赖包
Russ Cox给出极其简短的四句话说明:

在实际使用vendor中，要搞清楚vendor规则和相关注意点，避免采坑

关于vendor的规则

1: vendor不能直接放在src/下，必须放在src/xxx/包及其子包里面
2: 不能使用同一级中另一个包里面的vendor模块
3: 可以使用上一级(父级)的vendor

开启vendor模式的包查找顺序

1: 首先在本包的vendor中查询
2: 然后在父级的vendor中查询，循环直到src退出(src/里面的vendor不会去查询)
3: 然后查询gopath/src/
4: 最后查询goroot/src/

关于vendor的一点思考

golang终于有了自己的依赖管理机制,可以看到golang在不断的完善自己，满足人们的工程需要。
但引入vendor同样也引来来一些问题：
由于go的package是以路径组织的，在编译时，不同层次的vendor中相同的包会分别编译链接，
导致编译变慢，和运行文件变大。
要避免这个问题只能做一些规范：
1: 作为lib的包 不携带vendor更佳
2: 一个project内的所有vendor都集中在顶层vendor里面

最后

golang官方实现了vendor依赖机制管理，但要更好更方便的用到项目上还需要一些第三方工具：
1：glide (个人推荐)
2: godep
