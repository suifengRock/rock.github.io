git - 团队多人协作最佳实践

2017-07-16

git是一个伟大的发明。是一个很好的分布式版本控制工具。
但在团队没有一个很好地规范的情况，会产生很多不必要的冲突和麻烦。
尤其是git分支管理维护，没有一个很好的，团队一致的流程，那将是一个灾难。

git分支管理

分支(branch)在git中是一个非常重要的概念。
对于分支管理策略，大家普遍遵从git-flow流程。

结合图片，简单介绍一下不同分支的职责。

1.Production分支(Master)

这个分支是发布到生产环境的代码，这个分支只能从其他分支合并，不能在这个分支直接修改。

2.Develop分支

这个分支是主开发分支，包含所有要发布到下一个Release的代码，这个主要合并自其他分支，比如Feature分支。

3.Feature分支

Feature 分支主要用来开发一个新的功能，一旦开发完成，合并回Develop分支，并且进入下一个Release，Feature分支可以选择删除或者保留。

4.Release分支

当需要发布一个新Release的时候，基于Develop分支创建一个Release分支，Release分支在测试过程中可能会修改，完成Release后，合并到Master和Develop分支。

5.Hotfix分支

当在Production发现新的Bug时候，需要创建一个Hotfix分支, 完成Hotfix后，合并回Master和Develop分支，所以Hotfix的改动会进入下一个Release。

Git Flow使用原则

  • Master分支是线上稳定分支，Release通常用作测试分支，Develop分支是开发应用的主分支
  • 所有的功能开发都在Feature分支进行，然后合并到Develop分支
  • Release分支发布后出现问题，直接在Release分支修改，避免Develop分支代码污染

git团队提交流程

团队中保持统一的提交流程，有利于减少不必要的冲突和异常问题，减少误操作，便于新成员融入。
在团队协作中，通常用一个git仓库中心，便于统一管理和存储，一般团内内部自建gitlib，或者选用公共平台私有团队仓库(github,bicketbucketd等).

下面分享一个，个人经过这几年来总结而来的提交流程：

当已经把更改的文件add，并且commit本地，想合并提交到团队共同开发分支时,假设目前自己本地的开发分支feature/new ,共同开发分支为develop。

  • git check develop
  • git pull origin develop
  • git checkout feature/new
  • git rebase develop
  • git push origin feature/new
  • 在线上仓库中心开启一个MR(merge request)或者PR(pull request), 把新提交feature/new合并到develop

此流程可以解决git团队协作中几大问题：
1：提交导致分支网络一片混乱
2：多人同时提交，代码被覆盖问题
3：code review 和合并权限管理问题
